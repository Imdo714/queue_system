> # 1. Introduction
매년 반복되는 수강신청 기간의 극심한 트래픽 집중은 대학 행정 시스템의 고질적인 불안정성을 초래합니다. 본 프로젝트는 이러한 기술적 한계를 Redis 기반의 고가용성 대기열 설계를 통해 극복하고자 기획되었습니다. 지난 개념화(`Conceptualization`) 단계에서 비즈니스 목적과 시스템의 큰 틀을 정의했다면 

이번 보고서에서는 Use Case 분석을 통해 학생과 관리자, 그리고 시스템 스케줄러 간의 상호작용을 상세화하고, Domain 및 Application 분석을 통해 대기열 아키텍처의 핵심 구조를 정의합니다. 또한, 사용자가 자신의 대기 순번을 실시간으로 확인하며 전략적 판단을 내릴 수 있도록 합니다. UML을 활용한 정밀한 분석 과정을 통해, 시스템이 초당 5,000건의 요청 속에서도 어떻게 정합성을 유지하며 안정적으로 작동하는지 그 구체적인 메커니즘을 이해할 수 있을 것입니다.

> # 5. Glossary

| 용어 (Term) | 정의 (Definition) |
| :--- | :--- |
| **TPS** | Transaction Per Second의 약자로, 1초당 시스템이 처리하는 트랜잭션의 수. |
| Concurrency | 동시성. 여러 요청이 공유 자원(Redis/DB)에 동시에 접근하려는 상태. |
| **Throttling** | 유량 제어. 시스템의 부하를 방지하기 위해 요청의 처리 속도를 강제로 제한하는 기법. |
| Sorted Set | Redis의 데이터 구조(ZSET). 점수(Score)를 기준으로 값을 정렬하여 저장하는 대기열 구현의 핵심 요소. |
| Batch Registration | 배치 승인. 대기열에 진입한 요청을 실시간으로 DB에 반영하지 않고, 일정 주기마다 모아서 일괄 처리하는 방식. |
| **Zombie Traffic** | 좀비 트래픽. 사용자의 반복적인 새로고침(F5)으로 인해 발생하여 시스템 자원을 낭비하는 불필요한 요청. |
| **Atomic** | 원자성. 작업이 중간에 끊기지 않고 '전부 성공'하거나 '전부 실패'해야 하는 성질. |
| **Data Integrity** | 데이터 정합성. 데이터베이스 내의 데이터가 현실의 실제 값과 일치하고 모순이 없는 상태. |
| **Mapping Entity** | 관계 엔티티. 다대다(N:M) 관계를 일대다(1:N) 관계로 분리하기 위해 중간에 배치하는 엔티티(ex: Registration). |
