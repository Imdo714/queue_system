> # 1. Introduction
매년 반복되는 수강신청 기간의 극심한 트래픽 집중은 대학 행정 시스템의 고질적인 불안정성을 초래합니다. 본 프로젝트는 이러한 기술적 한계를 Redis 기반의 고가용성 대기열 설계를 통해 극복하고자 기획되었습니다. 지난 개념화(`Conceptualization`) 단계에서 비즈니스 목적과 시스템의 큰 틀을 정의했다면 

이번 보고서에서는 Use Case 분석을 통해 학생과 관리자, 그리고 시스템 스케줄러 간의 상호작용을 상세화하고, Domain 및 Application 분석을 통해 대기열 아키텍처의 핵심 구조를 정의합니다. 또한, 사용자가 자신의 대기 순번을 실시간으로 확인하며 전략적 판단을 내릴 수 있도록 합니다. UML을 활용한 정밀한 분석 과정을 통해, 시스템이 초당 5,000건의 요청 속에서도 어떻게 정합성을 유지하며 안정적으로 작동하는지 그 구체적인 메커니즘을 이해할 수 있을 것입니다.

> ## 2.2.1 Use Case #1: Register Member (회원가입)

| 항목 | 내용 |
| :--- | :--- |
| **Summary (요약)** | 신규 사용자가 학번 정보를 기반으로 시스템 계정을 생성하는 기능 |
| **Scope (범위)** | YU Queue System |
| **Level (수준)** | User level (사용자 수준) |
| **Author (작성자)** | 임도현 |
| **Last Update (최종 수정일)** | 2026-04-24 |
| **Status (상태)** | Analysis (분석) |
| **Primary Actor (주요 액터)** | Student (학생) |
| **Preconditions (사전 조건)** | 시스템에 등록되지 않은 신규 사용자여야 함 |
| **Trigger (트리거)** | 회원가입 페이지에서 정보 입력 후 '가입' 버튼 클릭 시 |
| **Success Post Condition (성공 후 조건)** | 사용자 정보가 DB에 저장되어 로그인이 가능한 상태가 됨 |
| **Failed Post Condition (실패 후 조건)** | 가입 실패 시 계정이 생성되지 않음 |

## MAIN SUCCESS SCENARIO (주요 성공 시나리오)

| Step (단계) | Action (행동) |
| :--- | :--- |
| 1 | 사용자가 회원가입 페이지에 접속한다. |
| 2 | 학번, 이름, 이메일, 비밀번호를 입력하고 가입 버튼을 누른다. |
| 3 | 시스템은 입력된 학번이 기존 데이터베이스에 존재하는지 확인한다. |
| 4 | 중복이 없음을 확인하고 사용자 정보를 데이터베이스에 저장한다. |
| 5 | 가입 성공 메시지를 띄우고 로그인 페이지로 전환한다. |
