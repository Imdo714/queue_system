> # 1. Business purpose

매년 수강신청 기간마다 대학가는 보이지 않는 전쟁터가 됩니다. 평상시보다 약 100배 이상의 트래픽이 단 1초 사이에 동시에 집중되면서 기존의 관계형 데이터베이스 기반 시스템은 한계에 직면합니다. 실제로 초당 10,000건 이상의 동시 접속 요청이 발생할 경우, 서버의 응답 시간은 평소보다 20배 이상 늘어나며 이 과정에서 서버 다운이나 접속 지연과 같은 치명적인 장애가 반복되고 있습니다. 이러한 불확실성은 사용자로 하여금 무분별한 새로고침을 시도하게 만들며, 이는 다시 서버 부하를 가중시키는 악순환으로 이어집니다. 본 프로젝트는 이러한 불안정한 환경을 기술적 수치로 제어하고 극복하기 위해 시작되었습니다.

현재 시스템의 가장 큰 문제점은 정보의 불투명성에서 오는 리소스 낭비입니다. 학생들은 단순히 로딩 화면을 바라보며 기다릴 뿐, 자신의 대기 순번이나 남은 좌석 정보를 실시간으로 확인하기 어렵습니다. 대기자가 정원보다 2배 이상 많은 상황에서도 사용자는 이를 인지하지 못한 채 평균 3분 이상의 무의미한 대기 시간을 소비하게 됩니다. 저희는 개발자 지망생으로서 이러한 정체 구간을 데이터로 시각화하고, 시스템이 감당할 수 있는 초당 처리량을 사전에 정의하여 전체 가용성을 확보할 수 있는 기술적 해결책을 제시하고자 합니다.

본 프로젝트의 최우선 목표는 고가용성 대기열 시스템의 구축을 통한 서비스 안정화입니다. 레디스 기반의 대기열 설계를 통해 초당 50,000건 이상의 요청을 유실 없이 수용하고, 메인 데이터베이스에는 초당 500건 수준의 안정적인 트래픽만 전달하도록 유량 제어를 실시합니다. 또한 사용자에게 1초 단위로 업데이트되는 실시간 대기 순번을 제공하여 본인의 신청 가능성을 즉각 판단할 수 있게 돕고, 가망 없는 대기를 스스로 철회하게 함으로써 유효 트래픽 처리 효율을 30% 이상 향상시키는 것을 목표로 합니다.

이 시스템의 일차적인 대상은 매 학기 수강신청 지연과 장애로 인해 불편을 겪는 전국 대학교의 학생들입니다. 또한 기존 시스템의 노후화로 인해 동시 접속자 처리에 한계를 느끼는 교육 기관 및 대학 행정 부처가 주요 타겟이 될 것입니다.

> # 3. Use case list

### 1. Membership (회원가입)
- **Actor**: Student
- **Description**: 시스템을 이용하기 위해 신규 회원은 자신의 정보를 등록한다. 학번, 성명, 이메일 등 본인 인증 및 수강 관리를 위한 필수 정보를 입력하여야 한다.

### 2. Login (로그인)
- **Actor**: Student, Administrator
- **Description**: 등록된 학번과 비밀번호를 통해 시스템에 접속한다. 인증 후 세션 또는 토큰을 발급받아 수강신청 및 관리 권한을 획득한다.

### 3. My Page (내 정보 조회)
- **Actor**: Student
- **Description**: 본인이 성공적으로 신청한 강의 목록을 확인하고, 현재 대기 중인 과목의 실시간 상태를 종합적으로 조회한다.

### 4. Search Course (강의 조회)
- **Actor**: Student
- **Description**: 현재 개설된 전체 강의 목록을 확인한다. 대기열에 저장된 실시간 데이터를 바탕으로 각 과목의 잔여 좌석 수를 즉각적으로 파악한다.

### 5. Enter Queue (대기열 진입)
- **Actor**: Student
- **Description**: 수강신청 버튼을 클릭하여 대기열에 진입한다. DB에 직접 부하를 주지 않고 Redis Sorted Set에 접속 기록을 남겨 자신의 대기 순번을 부여받는다.

### 6. Inform Rank (실시간 순번 조회)
- **Actor**: Student
- **Description**: 대기열 진입 후 내 앞에 남은 대기 인원을 실시간으로 확인한다. 시스템의 처리 속도에 따라 줄어드는 순번을 모니터링한다.

### 7. Strategic Cancel (전략적 취소)
- **Actor**: Student
- **Description**: 자신의 순번과 잔여석을 비교하여 가망이 없다고 판단될 경우 대기를 취소한다. 취소 즉시 대기열에서 제거되어 시스템 자원 낭비를 방지한다.

### 8. Batch Registration (배치 승인 처리)
- **Actor**: System (Scheduler)
- **Description**: 대기열 상위 사용자들을 일정 주기마다 추출하여 실제 DB에 수강 내역을 저장한다. 이를 통해 급격한 트래픽으로부터 메인 DB를 보호한다.

### 9. Check Duplicate (중복 검증)
- **Actor**: System
- **Description**: 실제 수강 처리 단계에서 동일 과목 중복 신청 여부나 시간표 중복 여부를 검증하여 데이터 정합성을 유지한다.

### 10. Result Notification (결과 알림)
- **Actor**: System
- **Description**: 배치가 완료되면 학생에게 최종 성공 여부 또는 실패 사유를 실시간으로 피드백한다.

### 11. Course & Capacity Management (강의 및 정원 관리)
- **Actor**: Administrator
- **Description**: 관리자는 신규 강의를 등록(`Registration`)하거나 폐강(`Termination`) 처리하며, 필요에 따라 실시간으로 수강 정원을 수정(`Adjustment`)한다.


> # 4. Concept of operation

### 1. Membership (회원 정보 등록)
* **Purpose**: 시스템 이용 권한 부여 및 사용자 식별
* **Approach**: 학번을 기반으로 신규 사용자를 등록한다. 성명, 이메일 등 필수 관리 정보를 입력받아 데이터베이스에 저장하며, 수강신청 시 본인 인증 및 데이터 정합성의 기초 자료로 활용한다.
* **Goals**: 유효한 학생 계정을 생성하여 시스템 접근 권한을 획득하도록 한다.

---

### 2. Login (인증 및 세션 관리)
* **Purpose**: 등록된 사용자의 신원 확인 및 보안 접속 유지
* **Approach**: 등록된 학번과 비밀번호를 검증하여 세션 또는 토큰을 발급한다. 학생과 관리자의 권한을 분리하여 각 역할에 맞는 인터페이스(대기열 진입 vs 강의 관리)를 제공한다.
* **Goals**: 인증된 사용자에게 수강신청 및 관리 기능을 안전하게 제공한다.

---

### 3. Search Course & Capacity (실시간 강의 조회)
* **Purpose**: 수강 가능 여부 판단을 위한 최신 정보 제공
* **Approach**: 전체 강의 목록과 함께 Redis Sorted Set에 저장된 실시간 잔여 좌석 데이터를 매칭하여 보여준다. 메인 DB의 부하를 최소화하기 위해 캐싱된 데이터를 우선적으로 참조한다.
* **Goals**: 학생이 잔여석을 확인하고 전략적으로 수강 과목을 선택할 수 있게 한다.

---

### 4. Enter Queue (고가용성 대기열 진입)
* **Purpose**: 폭발적인 트래픽으로부터 메인 시스템 보호 및 순서 보장
* **Approach**: 신청 버튼 클릭 시 DB에 즉시 기록하지 않고, Redis Sorted Set을 활용하여 요청 순서대로 대기열에 진입시킨다. 초당 수만 건의 요청을 메모리 기반으로 빠르게 수용한다.
* **Goals**: 서버 다운 없이 대량의 동시 접속 요청을 유실 없이 수용한다.

---

### 5. Inform Rank & Strategic Cancel (대기 상태 관리)
* **Purpose**: 대기 투명성 확보 및 유효 트래픽 효율화
* **Approach**: 대기열 내의 인덱스를 계산하여 사용자에게 실시간 순번을 1초 단위로 업데이트하여 제공한다. 순번이 너무 뒤처진 사용자가 스스로 대기를 취소할 수 있는 기능을 제공하여 불필요한 시스템 리소스 점유를 방지한다.
* **Goals**: 대기 시간을 시각화하여 사용자의 불안감을 해소하고 시스템 가용성을 높인다.

---

### 6. Batch Registration & Check Duplicate (배치 기반 수강 처리)
* **Purpose**: 안정적인 데이터 쓰기 및 무결성 유지
* **Approach**: 스케줄러가 대기열 상위 사용자를 일정 단위(Batch)로 추출하여 순차적으로 메인 DB에 기록한다. 처리 과정에서 중복 수강 및 시간표 겹침 여부를 최종 검증한다.
* **Goals**: 초당 처리량을 제어(Throttling)하여 DB 장애를 방지하고 데이터 정합성을 확보한다.

---

### 7. Result Notification (최종 결과 통보)
* **Purpose**: 수강신청 결과의 즉각적인 피드백
* **Approach**: 배치가 완료된 건에 대해 성공 또는 실패(정원 초과, 중복 등) 사유를 실시간 알림창이나 마이페이지 상태 업데이트를 통해 전달한다.
* **Goals**: 사용자가 신청 완료 여부를 즉시 확인하고 다음 행동을 결정할 수 있게 한다.

---

### 8. Course & Capacity Management (강의 및 정원 운영)
* **Purpose**: 학사 일정에 따른 유연한 강의 데이터 관리
* **Approach**: 관리자 권한으로 신규 강의를 등록하거나 폐강 처리한다. 특히 수강 수요에 따라 실시간으로 정원을 조정하며, 변경된 정원은 즉시 대기열 시스템의 판단 기준에 반영된다.
* **Goals**: 변화하는 학사 상황에 맞춰 실시간으로 수강 환경을 최적화한다.

> # 5. Problem statement

### Overview

본 프로젝트는 초당 5,000건의 진입 요청을 수용하고 300 TPS 수준으로 DB 부하를 제어하는 고가용성 수강신청 시스템을 지향한다. 단순히 "장애가 없어야 한다"는 추상적인 목표를 넘어, 실제 발생 가능한 기술적 병목 현상을 정의하고 이를 해결하고자 한다.

> ### 1) Problem #1
기존 RDBMS 중심의 아키텍처는 임계치 이상의 트래픽 유입 시 DB Connection Pool 고갈로 인해 시스템 전체 마비를 초래한다. 이는 단순한 '오류'를 넘어 서비스 가용성의 완전한 상실을 의미한다. 따라서 Redis를 완충 지대로 활용하여 DB 쓰기 요청을 초당 300건으로 서버가 감당할 수 있는 만큼만 넘겨(**Throttling**) 시스템 생존성을 보장해야 한다.

> ### 2) Problem #2
시스템 내부적으로는 Redis Sorted Set을 통해 중복 진입을 차단하지만, 사용자에게 실시간 상태를 제공하지 않을 경우 발생하는 반복적인 재요청 트래픽이 서버 자원을 낭비시킨다. 이는 네트워크 대역폭과 애플리케이션 쓰레드를 불필요하게 점유하여 실제 유효 트래픽 처리를 방해한다. 실시간 순번 제공을 통해 사용자의 심리적 불안감을 해소하고 불필요한 HTTP 요청을 억제해야 한다.

> ### 3) Problem #3
비동기 배치 처리 방식에서는 Redis 대기열 순서와 실제 DB의 잔여 좌석 간 불일치가 발생할 수 있다. 정원이 마감되는 시점에 대기열 상위권 사용자가 실패하거나, 중복 신청이 승인되는 등의 정합성 문제를 해결하기 위해 분산 환경에서의 원자적(Atomic) 처리와 최종 검증 로직 구현이 필수적이다.

> # 6. Glossary

본 문서에서 사용하는 주요 용어에 대한 정의는 다음과 같다.

- **TPS (Transactions Per Second)**: 초당 처리되는 트랜잭션의 수. 시스템의 처리 성능을 나타내는 지표로 활용된다.

- **Redis Sorted Set**: 키(Key) 하나에 여러 개의 요소(Member)를 점수(Score)와 함께 저장하며, 점수 순으로 자동 정렬되는 데이터 구조이다. 본 프로젝트에서 대기 순번을 결정하는 핵심 저장소로 사용된다.

- **Throttling (유량 제어)**: 시스템의 자원 고갈을 방지하기 위해 단위 시간당 처리되는 요청의 수를 강제로 제한하는 기술이다.

- **Data Integrity (데이터 정합성)**: 데이터베이스 내의 데이터가 현실 세계의 실제 값과 일치하고 모순이 없는 상태를 의미한다.

- **Batch Registration (배치 승인)**: 대기열에 진입한 대량의 요청을 일정 주기마다 묶어서 메인 데이터베이스에 일괄 반영하는 프로세스이다.

- **Atomic (원자성)**: 데이터베이스 트랜잭션의 특성 중 하나로, 일련의 연산들이 모두 성공하거나 모두 실패해야 함을 보장하여 데이터 오류를 방지한다.
